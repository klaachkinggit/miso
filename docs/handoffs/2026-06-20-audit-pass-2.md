## New / Clarified Findings

1. Checkout login loses the buyer's selected checkout target.
   `src/app/checkout/card/page.tsx:18` calls `requireUser()`, which redirects unauthenticated users to `/login` at `src/lib/auth.ts:30`. `src/app/login/actions.ts:35` redirects to role default after login and the login form carries no `next` parameter, so a buyer sent to login from `/checkout/card?...` loses the category/listing/quantity/promo context.

2. Checkout success page can display a pending message that will never refresh.
   `src/app/checkout/success/page.tsx:56` enables `StatusPoller` only when a valid payment is resolved, but `src/app/checkout/success/page.tsx:82` still tells buyers the page refreshes when `marketplace_payment_id` is missing or invalid.

3. Checkout success leaks internal status labels.
   `src/app/checkout/success/page.tsx:18` maps all non-terminal states to `fulfillment_pending`, and `src/app/checkout/success/page.tsx:68` renders that raw underscore status in the buyer-visible badge.

4. Stripe checkout error state can trap the buyer.
   `src/app/checkout/card/payment-form.tsx:161` displays an error and `src/app/checkout/card/payment-form.tsx:166` disables the pay button while `error` is truthy. There is no visible retry/reset path after a recoverable Stripe error.

5. Marketplace payment refund authorization trusts the first item to represent a multi-item payment.
   `src/app/admin/actions.ts:765` loads only one `marketplace_payment_items` row to derive the payment event. `supabase/migrations/20260613090000_marketplace_payment_items.sql:7` does not encode that all items for one payment belong to the same event. This requires data drift or service-role corruption, so it is lower than a direct IDOR, but the schema/action invariant is weak.

6. Unit fakes mask real Postgres constraints.
   `src/lib/stripe-marketplace/__tests__/helpers/fake-marketplace.ts:102` inserts rows into maps without enforcing unique constraints/checks. This hides first-pass issues like multi-item duplicate `provider_session_id`, and also misses `marketplace_payment_items.purchase_id unique` / non-negative amount constraints from `supabase/migrations/20260613090000_marketplace_payment_items.sql:11`.

7. External refund webhook has no behavioral test.
   `src/lib/stripe-marketplace/__tests__/webhook.test.ts:7` covers only `extractChargeId`; it does not exercise `charge.refunded`, cumulative refund selection, or `recordExternalChargeRefund()` behavior.

8. Manual marketplace refund tests miss linked purchase/ticket/listing lifecycle effects.
   Existing happy-path refund coverage checks payment/reversal outcome but not whether linked purchases or tickets move consistently after `src/lib/stripe-marketplace/refunds.ts:82` transitions a payment to `REFUNDED`.

9. Legacy ticket refund treats `refund_pending` as refundable.
   UI only blocks `used` and `refunded` at `src/app/admin/events/[id]/refunds-panel.tsx:67`; backend only rejects `refunded` and `used` at `src/lib/refunds/refund.ts:26`. A cancellation-created `refund_pending` ticket can be submitted again through this path.

10. Public/buyer copy still overexposes chain/wallet language.
    Examples: `src/components/landing/proof-strip.tsx:11` says `On-chain tickets`; `src/components/landing/capabilities.tsx:26` says `ERC-721 transfer`; `src/app/marketplace/page.tsx:17` says anti-scalping is enforced on-chain; `src/app/tickets/page.tsx:62` labels the normal tickets page `Buyer · Wallet`; `src/components/tickets/ticket-card.tsx:113` displays `Wallet`. This conflicts with `docs/CONTEXT.md` guidance to keep normal buyer UI on Digital ticket language unless in advanced proof/export flows.

11. Public legal pages still expose placeholder final-policy/final-terms copy.
    Same as pass 1, confirmed rendered with `agent-browser`: `src/app/legal/terms/page.tsx:12`, `src/app/legal/privacy/page.tsx:12`, `src/app/legal/cookies/page.tsx:12`.

12. Landing storefront section disappears when no active orgs exist.
    `src/components/landing/storefronts.tsx:18` returns `null`; there is no empty/fallback section. This is not a correctness bug, but it creates an abrupt page rhythm in empty/local data.

## Highest-Risk Findings

1. `src/app/admin/actions.ts:428` has a weaker guard than `src/app/admin/site/page.tsx:12`.
   The page requires `requireRole("admin")`, but `updateSiteSettings` only calls `requireOrganizerWorkspace()`, so an organizer could invoke the action directly and mutate global landing artwork.

2. `src/lib/stripe-marketplace/payments.ts:662` does not reapply public resale sellability checks before checkout.
   Public listing code filters on listed ticket, published event, future date, active organization, and resale-enabled category in `src/lib/marketplace/public.ts:34`; direct checkout only claims an active listing.

3. `supabase/migrations/20260605004939_organization_foundation.sql:493` grants org admins `for all` on `events`.
   That includes chain authority columns such as `nft_contract_address` and `role_admin_address`, which later feed privileged backend wallet writes in `src/lib/tickets/lifecycle.ts:413`, `src/lib/resale/listing.ts:419`, and `src/lib/verification/redeem.ts:312`.

4. Multi-item primary checkout likely violates the old unique constraint renamed from `stripe_checkout_session_id` to `provider_session_id`.
   The constraint originates at `supabase/migrations/0001_init.sql:137`, is renamed in `supabase/migrations/0005_payments_refactor.sql:11`, and current checkout writes the same idempotency key to every purchase at `src/lib/stripe-marketplace/payments.ts:457`.

5. Canceled primary marketplace payments do not release current reservations.
   Current primary payments use `purchase_id: null` and `marketplace_payment_items` at `src/lib/stripe-marketplace/payments.ts:522`, but terminal failure inventory release only handles primary payments with `payment.purchase_id` at `src/lib/stripe-marketplace/state-machine.ts:184`.

6. Future organizer signup onboarding can create Stripe accounts not represented in `stripe_seller_accounts`.
   `/signup/organizer/stripe` uses legacy `src/lib/payments/stripe-connect.ts:133`, while marketplace readiness checks `stripe_seller_accounts` via `assertPayoutReady()` at `src/lib/stripe-marketplace/payments.ts:455`. The migration `20260612221500_backfill_stripe_seller_accounts.sql` only backfills existing profiles once.

7. Legacy refund code treats `provider_session_id` as a Stripe Checkout Session id.
   `src/lib/refunds/refund.ts:55` passes it to `refundStripeSession()`, but marketplace purchases store checkout idempotency there and PaymentIntent ids in `provider_payment_id`.

8. Marketplace refunds do not move ticket/purchase/listing rows through the documented refund lifecycle.
   `src/lib/stripe-marketplace/refunds.ts:59` transitions `marketplace_payments`, while `REFUNDED` in `src/lib/stripe-marketplace/state-machine.ts:93` only updates the payment row.

9. External Stripe refunds are not cumulative.
   `src/lib/stripe-marketplace/webhook.ts:105` reads `charge.refunds.data[0]`; `src/lib/stripe-marketplace/refunds.ts:145` then compares that single refund amount while the comment claims cumulative handling.
